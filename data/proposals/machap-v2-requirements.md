**作成日**: 2026-03-05
**対象**: machap-portal/index.html
**ユーザー**: サノ・中・ししょー（3人チーム）
**方針**: スマホファースト / 3人の使いやすさMAX / ビジネスポータルとしての完成度

## 1. 現状分析

### 1.1 現在のアーキテクチャ

| 項目 | 現状 |
|------|------|
| データ保存 | localStorage（`machap_portal_data`キー） |
| フレームワーク | バニラHTML/CSS/JS（単一ファイル 2700行） |
| レスポンシブ | 768px以下でサイドバー折りたたみ |
| viewport | `width=device-width, initial-scale=1.0`（ピンチズーム制御なし） |
| AI連携 | キーワードマッチングのみ（Gemini未使用） |
| データ共有 | なし（端末ごとにlocalStorage分離） |

### 1.2 各STEPの現在の機能と課題

#### FIND（課題・アイデア発見）
- **実装済み**: クイック投稿、カテゴリ管理、ステータス別/カテゴリ別ビュー切替、投票（サ/中/師）、優先度・ステータス管理、キーワードベース自動分類
- **課題**:
  - アイデア同士の関連性が見えない（独立したカードの羅列）
  - 影響度×実現容易度マトリクスが未実装（deep-designで計画済みだが未着手）
  - 他STEPとの接続が「Project化」ボタンのみ（FIND→MAKEの一方向だけ）
  - FINDに投稿すべき具体的アイデア群がまだ登録されていない

#### MAKE（プロダクト化）
- **実装済み**: タスクトラッカー（担当・ステータス管理）、プロダクト一覧、Sprint週次バー
- **課題**:
  - タスクとプロダクトが別管理で、連動が弱い
  - プロダクトのSTEP間進捗（FIND/MAKE/OFFER/PRESENT/CONSULT）が見えない
  - スプリントが「今週の進捗バー」だけで、タスクとの紐付けがない

#### OFFER（提供管理）
- **実装済み**: パイプライン（公開済み→FB中→完了）、提供先管理、フィードバック記録
- **課題**:
  - FINDやMAKEのプロダクトとの紐付けが `projectId` ベースだがUIで選択しづらい
  - 提供先（=対象コミュニティ/個人）のデータベースがない
  - フィードバックからFINDへの自動還元リンクがない

#### PRESENT（発信管理）
- **実装済み**: 発信プラン（プロダクト×チャネルのマトリクス）、カレンダー、テンプレート＆下書き生成、発信ログ、チャネル別サマリー
- **課題**:
  - テンプレート変数展開が手動（Gemini活用の自動生成が未実装）
  - 発信プランとプロダクトの関係がドロップダウン選択のみ

#### CONSULT（相談管理）
- **実装済み**: 案件リスト、収益サマリー
- **課題**:
  - パイプライン（受付→ヒアリング→提案→実行中→完了）のカンバンビューがない
  - 収益配分計算（30/30/40）がUIに組み込まれていない
  - 相談→新規FINDアイデアへの変換ボタンがない

#### 横断機能
- **実装済み**: Dashboard（統計サマリー＋アクティビティタイムライン）、Phase Progress（Phase 1-3チェックボックス）、Export/Import JSON
- **課題**:
  - **ライン接続がない** — 1つのアイデアがFIND→MAKE→OFFER→PRESENT→CONSULTと流れる一気通貫のトラッキングが存在しない
  - プロジェクト詳細ビュー（5STEP進捗の1画面確認）が未実装
  - データがlocalStorageのみで3人間の共有ができない

## 2. ライン接続設計 — アイデアからプロダクトの一気通貫フロー

### 2.1 コアコンセプト: 「ライン」

1つのアイデアが5STEPを流れる一連の軌跡を **ライン（Line）** と呼ぶ。

```
[FIND: アイデア登録]
    ↓ 採用
[MAKE: プロダクト化・開発]
    ↓ 完成
[OFFER: 提供・公開]
    ↓ 反響
[PRESENT: 発信]
    ↓ 認知
[CONSULT: 相談が来る]
    ↓ フィードバック
[FIND: 次のアイデア] ← サイクル
```

#### 途中STEPからのスタートも可能
- MAKEから: 「もう作り始めてる」→ MAKE登録時にFINDも自動生成（後付けで経緯を記録）
- OFFERから: 「もう使ってもらってる」→ OFFER登録時にFIND+MAKEをスキップマーク
- CONSULTから: 「相談が先に来た」→ CONSULT登録→FINDアイデアへ還元

### 2.2 データモデル（拡張）

```javascript
// 現行のStore構造を拡張

  ideas: [],       // FIND
  tasks: [],       // MAKE
  offers: [],      // OFFER

  categories: [],

  activity: [],

  // --- v2追加 ---

// Line（ライン）
{
  id: 'line_xxxxx',
  title: 'マルタ村ポータル横展開',  // ライン全体の名称

  ideaId: 'idea_xxxxx',              // FIND のアイデアID（null可）
  productId: 'prod_xxxxx',           // MAKE のプロダクトID（null可）
  offerIds: ['off_xxxxx'],           // OFFER のID群
  pubPlanIds: ['pub_xxxxx'],         // PRESENT の発信プランID群

{

    trend: 2,                 // トレンド度 (1-5)

  }

```

## 3. AI機能設計

### 3.1 技術方針

- **API**: Vercel Proxy経由のGemini API（`/api/gemini`、gemini-2.5-flash）
- **コスト**: 無料枠内で収まるよう、リクエスト頻度を制御
- **フォールバック**: APIエラー時はキーワードマッチング（現行ロジック）にフォールバック

### 3.2 AI優先度判断機能

#### 実装方針
FINDに登録された全アイデアを定期的（またはボタン押下時）にGeminiに送り、以下を算出する。

```
プロンプト設計:
「以下はチーム"MACHAP"のアイデアリストです。
チームは3人（サノ: 全体管理・コンセプト、中: プロダクト開発、ししょー: 意見・拡散）。
モルック関連のコミュニティ支援プロダクトを中心に活動しています。

各アイデアについて以下を1-5で評価してください:
- impact: ターゲットへの影響度
- feasibility: 実現容易度（技術的ハードル、工数）
- trend: トレンド度（今やるべきか）

また、以下を提案してください:
- priority_order: 推奨着手順位
- suggested_assignee: 最適な担当者（サノ/中/ししょー）
- related_pairs: 親和性の高いアイデアのペア
- reasoning: 判断理由（1-2行）

アイデアリスト:
[JSONで全アイデアを渡す]
」
```

#### UI: AIアドバイザーパネル
```
┌──────────────────────────────────────┐
│ AI Advisor                   [更新]  │
├──────────────────────────────────────┤
│ 推奨着手順位:                         │
│  1. チーム名作成補助 (impact:5, easy:4)│
│     → 中さん向き。1日で作れる。        │
│     モルック大会の定番ニーズ           │
│                                      │
│  2. エントリーフォーム (impact:4, easy:3)│
│     → サノ向き。大会運営の核になる     │
│                                      │
│ 親和性ペア:                           │
│  [チーム名作成補助] ←→ [全国チーム登録] │
│  [会場版SNS] ←→ [すれ違い対戦機能]     │
│  [大会集計] ←→ [エントリーフォーム]     │
│                                      │
│ トレンド提案:                         │
│  「モルック × AI」が最近のバズワード。   │
│  チーム名作成補助にAI機能をつけると     │
│  話題性が高い。                       │
└──────────────────────────────────────┘
```

### 3.3 親和性分析

- アイデア同士の関連度をGeminiで判定
- 結果を `relatedIdeaIds` / `relatedLineIds` に保存
- UIでは「関連アイデア」としてカード下部に表示
- タップで関連アイデアにジャンプ可能

### 3.4 トレンド提案

- ボタン押下時にGeminiへ「モルック + コミュニティ + テクノロジー」のトレンド分析を依頼
- 結果をFINDのトップに「AIトレンド提案」として表示
- 具体例: 「モルック×ライブ配信が海外で流行」→ ライブ配信用表示のアイデアを後押し

### 3.5 自動分類の強化

現行のキーワードマッチングをGemini APIに置き換える。

```
現行: CATEGORY_KEYWORDS / STEP_KEYWORDS による文字列マッチ
v2:  Gemini APIで文脈を理解した分類
     - カテゴリ判定精度の大幅向上
     - 新カテゴリの自動提案
     - STEPの文脈判定（「〇〇を作りたい」vs「〇〇を作った」の区別）
```

- **フォント統一**: 数字はタブラー（等幅）、日本語は游ゴシック/Noto Sans JP

## 5. データ共有戦略

### 5.1 現状の問題

localStorageは端末ごとに独立しているため、サノ・中・ししょーの3人で同じデータを見られない。

### 5.2 段階的アプローチ

#### Phase A: Export/Import改善（v2.0で実装）
- ワンタップでJSONエクスポート→LINEで共有→相手がインポート
- 差分マージ機能（同一IDのデータは新しい方を採用）
- 「最終同期日時」の表示

#### Phase B: Firebase RTDB連携（v2.1で実装）
- 既存のFirebase基盤（viisi-masterと同じプロジェクト、または専用プロジェクト）を活用
- パス: `/machap-portal/` 以下にデータ保存
- 認証: Google Auth（3人のGoogleアカウント）
- リアルタイム同期で3人が同時に同じデータを閲覧・編集
- オフライン対応: localStorageをキャッシュとして併用

## 8. 実装上の注意点

### 8.1 単一ファイル維持 vs 分割

現行は単一HTML（2700行）。v2で機能が大幅に増えるため、以下を検討:

- **推奨**: 単一ファイル維持（GitHub Pagesでの配信簡便性、既存運用との一貫性）
- **代替**: CSS/JSを外部ファイル化（machap-portal/style.css, machap-portal/app.js）
- 判断基準: 3000行を超える場合は分割を検討

### 8.2 localStorage容量

- localStorageの上限は通常5MB
- アイデア100件 + タスク100件 + その他データ = 推定200KB程度
- AI提案履歴を無制限に保存すると肥大化するため、直近50件に制限

### 8.3 後方互換性

- v1のlocalStorageデータをv2で読み込めるようにマイグレーション関数を実装（既存の `migrateStore()` パターンを踏襲）
- 新フィールド（`lines`, `aiScore` 等）はオプショナルとし、未設定でも動作するように

### 8.4 パフォーマンス

- Gemini APIコールはデバウンス（連続クイック投稿時に1回にまとめる）
- DOM更新は差分のみ（全再描画を避ける）
- カレンダーやマトリクスのレンダリングはrequestAnimationFrame使用
