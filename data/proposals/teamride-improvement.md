### 1.3 自動配車アルゴリズムの問題点

現在の `runAutoAssign()` 関数（L1881-2003）の分析結果:

**問題1: 同一エリアマッチングのロジック欠陥**
```
現在の処理順:
1. 機材割り当て（ドライバーの空席を減らす）
2. ドライバーを空席降順ソート
3. ライダーを必要席降順ソート
4. 同一エリア内でマッチング
5. エリア跨ぎでマッチング
```
- ゾーン内でドライバーを巡回する際、`zoneDrivers.find()` で最初に見つかった空きドライバーに詰め込む
- 大型車の席を先に消費し、小型車が残る非効率パターンが発生する
- 例: 7人乗り1台、5人乗り1台のゾーンに3人+2人のライダー → 7人乗りに全員詰め込み、5人乗りが空のまま

**問題2: 備品考慮の不完全性**
- 備品を持つドライバーの席を減らすが、「備品持ちの車を優先的に使う」判断がない
- 備品を持っていないドライバーの車に備品を載せる考慮がゼロ
- 備品が席を圧迫してもそのドライバーに乗客を割り当てようとする

**問題3: 距離/住所ベースの組み合わせが未実装**
- エリア（ゾーン）でのグルーピングはあるが、同一エリア内での距離最適化がない
- メンバーに住所/座標データが存在しない

**問題4: 台数制限の考慮がない**
- 「制限台数」の概念がなく、車を出すと言った全員がドライバーになる
- 実際には「車は出せるけど今回は出さなくていい」ケースの制御がない

## 3. 自動配車アルゴリズム再設計

### 3.1 設計思想

配車の最適化は **3軸スコアリング方式** で行う。各ドライバーと各ライダーの組み合わせにスコアを付け、総合スコアが最大になる割り当てを採用する。

### 3.2 スコアリング基準

#### 軸1: 備品スコア（最優先）

| 条件 | スコア |
|------|--------|
| イベントに必要な備品を持つドライバー → 備品を積む余地がある | +50 |
| 備品を積むと席が減るが、それでも乗客を乗せられる | +30 |
| 備品を積むと席がゼロになる（備品専用車） | +10（乗客は割り当てない） |

**ロジック:**
1. イベントの必要備品リストを取得
2. 備品を持つドライバーを特定
3. 備品の「占有席数」を差し引いた残席を計算
4. 備品持ちドライバーは最優先で使用する（車を出すかどうかの判断に影響）

#### 軸2: 乗車人数スコア（効率性）

| 条件 | スコア |
|------|--------|
| 1台で多くの人を運べる（残席が多い） | +20 × (残席数 / 最大残席数) |
| 弟妹連れのライダーを大型車に優先割り当て | +15 |
| 小型車には子供1人だけのライダーを優先 | +10 |
| 台数制限内で最少台数に収まる組み合わせ | +25 |

**台数制限の扱い:**
- イベント作成時に「使用台数上限」を設定可能にする
- 上限を超える場合、空席が最も少ないドライバーを「乗せてもらう側」に自動転換
- 車を出す意思があっても、台数制限で不要ならスキップする

#### 軸3: 距離スコア（近接性）

| 条件 | スコア |
|------|--------|
| 同一エリア内のマッチング | +30 |
| 同一エリアかつ住所が近い（直線距離500m以内） | +40 |
| 同一エリアかつ住所が近い（直線距離1km以内） | +35 |
| 隣接エリアとのマッチング | +15 |
| エリアが遠い | +0 |

**住所/座標の扱い:**
- メンバー登録時に「おおよその住所」を入力可能にする（番地不要、町名レベルでOK）
- 住所から緯度経度を取得（Geocoding API、初回登録時のみ）
- 座標未設定の場合はエリア単位でのスコアリングにフォールバック
- プライバシー配慮: 「○○丁目」レベルまでしか保存しない

### 3.3 アルゴリズムフロー

```
入力:
  - drivers[]: ドライバー一覧（空席数、備品、座標）
  - riders[]: ライダー一覧（必要席数、座標）
  - equipment_needed: イベントの必要備品
  - max_cars: 台数制限（任意）

Step 1: 備品割り当て
  FOR each 必要備品:
    備品を持つドライバーをスコア順でソート
    最適なドライバーに備品を割り当て
    ドライバーの空席を備品分だけ減算

Step 2: 台数制限チェック
  IF max_cars が設定されている AND drivers.length > max_cars:
    空席が少ない順にドライバーをライダーに転換
    （備品割り当て済みのドライバーは除外）

Step 3: スコア行列の構築
  FOR each driver × rider の組み合わせ:
    score = 備品スコア + 乗車人数スコア + 距離スコア
    IF driver.空席 < rider.必要席数: score = -Infinity

Step 4: 貪欲法による割り当て
  WHILE 未割り当てライダーが存在:
    スコア行列から最高スコアの(driver, rider)ペアを選択
    割り当て実行、ドライバーの空席を更新
    行列を再計算

Step 5: 検証
  未割り当てライダーがいれば警告
  同一エリアにドライバー不在なら警告
  備品不足があれば警告

出力:
  - assignment: ドライバー別の乗客リスト
  - warnings: 警告メッセージ一覧
  - score_breakdown: スコア内訳（デバッグ/管理者向け）
```

### 3.4 データモデル拡張

```
members/{member_id}/
  + address: string          # 町名レベル（例: "世田谷区三軒茶屋"）
  + lat: number              # 緯度（Geocodingで自動取得）
  + lng: number              # 経度（Geocodingで自動取得）

events/{event_id}/
  + max_cars: number | null  # 台数制限（null=無制限）
```

## 5. UI/UX デザイン方針

### 5.1 デザインコンセプト

**「ポップ & シンプル」** — 少年サッカーの元気さを反映したカラフルで明るいUI

| 要素 | 方針 |
|------|------|
| カラーパレット | サッカーグリーン基調は維持、アクセントにオレンジ・イエロー・スカイブルー |
| フォント | 現状の system-ui を維持（読みやすさ最優先） |
| アイコン | 絵文字ベースを継続（追加のフォント読み込み不要） |
| 角丸 | 現状の14pxを維持（親しみやすさ） |
| アニメーション | 軽量なtransition（0.15-0.25s）、過剰なアニメは排除 |
| 配色バランス | 白背景ベース、カード影は控えめ、色はアクセントのみ |

## 6. 技術的実装方針

### 6.1 アーキテクチャ

現在の **単一HTML + Firebase RTDB** のアーキテクチャは維持する。理由:
- GitHub Pagesで無料ホスティング
- サーバーサイド不要
- チーム内利用（100人以下）にはFirebase無料枠で十分
- 単一HTMLは共有が簡単（URLを1つ送るだけ）

### 6.2 Geocoding（住所→座標変換）

**方針: 無料で済ませる**

| 選択肢 | 判定 | 理由 |
|--------|------|------|
| Google Geocoding API | 不採用 | 有料（$5/1000リクエスト）、APIキー管理が必要 |
| 国土地理院API | 採用候補 | 無料、日本国内に限定されるが十分 |
| 手動座標入力 | フォールバック | 地図上でピンを立てる方式 |
| OpenStreetMap Nominatim | 採用候補 | 無料、利用規約の範囲内（1秒に1リクエスト制限） |

**推奨: 住所入力時にOpenStreetMap Nominatimで座標取得 + 失敗時は手動ピン**

### 6.3 LINE連携

**現在:** テキストをクリップボードにコピー → LINEに手動ペースト
**改善案:**

| レベル | 実装 | 工数 |
|--------|------|------|
| Level 1 | LINE URLスキームで共有（`line://msg/text/...`） | 30分 |
| Level 2 | LINE LIFF SDK導入（アプリ内からLINEに直接送信） | 2-3時間 |
| Level 3 | LINE Messaging API（Bot経由で自動配信） | 1-2日 |

**推奨: Level 1から開始し、要望に応じてLevel 2へ**

### 6.4 Instagram連携

| 方式 | 内容 |
|------|------|
| 結果カード画像生成 | html2canvasで大会結果をPNG画像化 |
| ストーリーシェア | Web Share API（`navigator.share`）で画像をInstagramストーリーに共有 |
| フィード投稿 | Instagram Graph APIは事業者向けのため不採用。画像保存→手動投稿 |

### 6.6 セキュリティ

| 項目 | 現状 | 改善 |
|------|------|------|
| 管理者認証 | 4桁PIN（平文でFirebaseに保存） | SHA-256ハッシュ化して保存 |
| Firebase Rules | なし（完全公開状態） | 読み取りは全員OK、書き込みは特定パスのみ |

## 付録: 配車シミュレーション例

### 入力条件
- メンバー12名（3エリア）
- ドライバー4名（うちテント持ち1名）
- 台数制限: 3台
- テント1張り必要

### 現行アルゴリズムの結果（問題あり）
```
🚗 佐藤家（テント持ち・8人乗り）→ テント積載で残3席 → 近くの3人を割当
🚗 山田家（7人乗り）→ 遠いエリアの5人を割当 ← 距離無視
🚗 高橋家（5人乗り）→ 空席余り
🚗 伊藤家 → 台数制限で不要だが車を出してしまう
⚠ 非効率: 4台使用（制限3台を超過）、山田家に遠いエリアの子が集中
```

### 改善後アルゴリズムの結果
```
🚗 佐藤家（テント持ち・8人乗り）→ テント積載残3席 → 同エリアの3人
🚗 山田家（7人乗り）→ 同エリアの4人（弟妹連れ含む）
🚗 高橋家（5人乗り）→ 残りエリアの2人
🏃 伊藤家 → 台数制限により「乗せてもらう」に自動転換 → 高橋家に同乗
✅ 3台で全員カバー、エリア内マッチング最適化
```
