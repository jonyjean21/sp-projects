# SP Portal / モルハブポータル 再構成設計書

## 現状の問題

SP Portalは「SPの全てを管理する場所」を目指しているが、実態はモルハブ管理機能（記事・大会・Xポスト）がメインで、SP個人のプロダクト管理やビルド履歴は付け足し感がある。

さらに、モルハブ関連だけでも以下のツールが分散している：
- **SP Portal**: 記事管理、大会キュー、大会データ、Xポスト、アナリティクス
- **hub-dashboard**: 向上委員会ダッシュボード（KPI表示、GA4/SC連携済み）
- **molkky-admin**: レガシー管理パネル
- **naimol-admin**: 大会結果→WP投稿
- **championship-admin**: 日本選手権トラッカー管理

また大会セクションが「大会情報」（キュー処理）と「大会データ」（WPビューア）に分裂している。

---

## 再構成方針

### SP Portal = SPの全体ダッシュボード（シンプルに）
- **プロダクト一覧**（カテゴリ別グリッド、ステータスバッジ付き）
- **ビルド履歴**（タイムライン形式）
- **クイックリンク**（各ポータル・外部サービスへ）
- ※WP記事数・大会数などモルハブ固有KPIは**不要**
- ※WP最新投稿などのアクティビティも**不要**

### モルハブポータル = モルハブの全て（新規・独立）
- ダッシュボード（KPI: 記事数・大会数・GA4セッション・SC等 ← hub-dashboard統合）
- 記事管理
- 大会管理（キュー + 全データ統合 + 自動検証 + 手動編集 + ソース表示）
- Xポスト生成
- アナリティクス（GA4/SC ← hub-dashboardで既に連携済み、これを統合）
- パートナー管理
- 大会結果（ナイモル）
- 選手権トラッカー
- 自動化パイプライン状態

### 独立を維持するもの
- **MACHAPポータル**: チーム3人共有。SP専用ではない
- **めぐり帳**: 完全に個人用
- **レシピBOX**: 家族用
- **公開ページ群**: championship/, viisi-master/ 等

※ **パートナーLP / partner-portal** はモルハブの活動なので、モルハブポータルに含める

## モルハブポータル 新構成（`molkky-portal/` 新規作成）

### サイドバー

```
Overview
  Dashboard（★hub-dashboard統合: KPI + GA4 + SC + SNS + WPデータ）

Content
  記事管理（一覧・作成・編集）
  Xポスト生成（promote + repost統合）

Tournament
  大会管理（★統合版：キュー + 全データ + 検証 + 編集 + ソース表示）

Analytics
  トラフィック（GA4連携 ← hub-dashboardから移植、連携済み）
  コンテンツパフォーマンス
  Search Console（← hub-dashboardから移植、連携済み）

Partners
  パートナー管理
  パートナーLP（partner/）
  パートナーポータル（partner-portal/）

Operations
  大会結果（ナイモル統合）
  選手権トラッカー（管理側）
  自動化パイプライン

Settings
  WordPress接続
  データ管理
```

### Dashboard — hub-dashboard統合

**方針**: 現在のhub-dashboard（向上委員会ダッシュボード）の全機能をモルハブポータルのダッシュボードに統合する。hub-dashboardで既にGA4・Search Console連携済みなので、そのコードを移植・統合。

含める内容:
- WP記事数・大会情報数（WP API自動取得）
- GA4セッション数・PV（API連携済み）
- Search Consoleデータ（API連携済み）
- SNS統計（X フォロワー等）
- パートナー数（Firebase自動取得）
- 会議アジェンダ
- クイックリンク（GA4・SC・WP管理画面）

統合後、hub-dashboard は廃止（リダイレクト）。

### 大会管理 — 統合設計（最重要）

**現状の問題**: 「キュー処理」と「データ閲覧」が別セクション

#### 基本方針
1. **自動処理が基本**: IFTTT/GAS → Firebase → 自動WP投稿をどんどん回す
2. **自動検証で品質担保**: 都道府県・開催日の抜けを自動検出し警告バッジ表示
3. **手動編集は検証後**: 自動検証で拾えない問題を人間が修正
4. **ソース情報の可視化**: X投稿の元テキスト・画像が見える状態を必ず作る

#### 統合UI設計

```
┌─────────────────────────────────────────────┐
│ 大会管理                          [手動追加] │
├─────────────────────────────────────────────┤
│ タブ: [全件] [未処理] [要確認] [最近の投稿]  │
├─────────────────────────────────────────────┤
│ 検索: [________] 地域:[▼] 月:[▼] 状態:[▼]  │
├─────────────────────────────────────────────┤
│ ┌───────────────────────────────────────┐   │
│ │ 🟡 PENDING  モルック大会 in 東京       │   │
│ │ 2026/04/15 東京 | IFTTT | → [処理]   │   │
│ └───────────────────────────────────────┘   │
│ ┌───────────────────────────────────────┐   │
│ │ ⚠️ WARNING  モルック体験会 千葉        │   │
│ │ 2026/03/20 千葉 | 公開中 | 規模なし    │   │
│ └───────────────────────────────────────┘   │
│ ┌───────────────────────────────────────┐   │
│ │ ✅ OK  第5回〇〇モルック大会            │   │
│ │ 2026/05/01 大阪 | 公開中 | 50チーム     │   │
│ └───────────────────────────────────────┘   │
│                                             │
│ ← 1 2 3 ... 28 →                           │
└─────────────────────────────────────────────┘
```

**タブの意味**:
- **全件**: WP投稿済み全大会 + Firebase未処理キュー = 全てがここに
- **未処理**: Firebaseキューの pending のみ（IFTTT/GAS/手動）
- **要確認**: 自動検証でwarningが付いた大会（都道府県なし・開催日なし等）
- **最近の投稿**: 直近7日間の投稿

**ソース可視化のポイント**:
- X投稿の元テキスト全文を表示（Gemini抽出前の生データ）
- **添付画像を表示**（写真に日時・場所等の情報が含まれていることが多い）
  - IFTTTが取得した画像URL or Xの画像URLを保存・表示
  - タップで拡大表示（モーダル）
- 元URLへのリンク（X投稿を直接確認できる）
- どのルートで収集されたか（IFTTT/GAS/手動）

#### 自動検証チェッカー
- 開催日が空 or 過去すぎる（1年以上前）→ ⚠️
- 都道府県が未設定 → ⚠️
- タイトルに「モルック」が含まれない（誤収集の可能性）→ ⚠️
- 重複疑い検出（同日・同県の大会）→ ⚠️
- 規模（参加チーム数等）が未設定 → ℹ️ (info)
- 警告バッジ付きで一覧に表示
- 「要確認」タブで問題ありデータだけ表示

#### 手動追加
- URLペースト → Gemini抽出 → フォーム自動入力 → 保存
- 空フォームからの手入力も可

#### 統計サイドパネル or 上部
- 月別投稿数の棒グラフ
- 都道府県別ヒートマップ
- ソース別割合（IFTTT/GAS/手動）
- 投稿日ベースの時系列管理

### データ永続化方針

**Firebaseに積極的に入れるもの**:
- 大会キュー（/tournament-queue ← 既存）
- チャプ会キュー（/chapche-queue ← 既存）
- パートナーデータ（/partner ← 既存）
- 大会の自動検証結果・警告ステータス（Firebase推奨）
- 大会のソース情報（元ツイートテキスト・画像URL）
- アナリティクスのスナップショット（定期取得したGA4/SCデータ）

**localStorageで良いもの**:
- WP認証情報
- UI設定（表示件数、フィルタ状態等）
- 一時的なキャッシュ

## 移行計画

### Phase 1: モルハブポータル構築
1. `molkky-portal/index.html` 新規作成
2. SP Portalからモルハブ機能を移植（記事・大会・Xポスト・アナリティクス）
3. hub-dashboardのKPI・GA4・SC連携コードを統合
4. 大会セクションを統合版に再設計（ソース可視化・画像表示含む）
5. IFTTTの大会収集時にソース情報（元テキスト・画像URL）もFirebaseに保存

### Phase 2: SP Portal軽量化
1. モルハブ機能を全て削除
2. ダッシュボードをプロダクト一覧 + ビルド履歴 + リンク集のみに
3. 各ポータルへのハブリンク追加

### Phase 3: レガシー整理
1. molkky-admin → モルハブポータルに吸収（廃止）
2. hub-dashboard → モルハブポータルに吸収（廃止 or リダイレクト）
3. apps/index.html → SP Portalのダッシュボードが代替（廃止 or リダイレクト）

## 技術方針
- 全て単一HTML（GitHub Pages、ビルド不要）
- localStorage永続化（WP認証は共有キーで横断）
- WP REST API（SiteGuard WAF対策: POSTのみ）
- Firebase RTDB（キュー・チャプ会・選手権 + ソース情報・検証結果）
- Vercel API Proxy（Gemini・Drive）
- X画像表示: FxTwitter API or X syndication APIで画像URL取得
