# 寄稿プロジェクト フロー設計メモ

> 作成日: 2026-02-24 / 更新: 2026-02-24

## 現状

- 連絡手段が寄稿者ごとにバラバラ（LINE、DM等）
- **現時点では大きな不便は感じていない**（寄稿者3名 + NICE MOLKKY JAPAN）
- 入稿→WordPress転記が手動 → PC時間が取れないとボトルネックに
- 寄稿者が今後増える見込み → 増えたときに備えてフローを設計しておく

## 選択肢の比較

| 方法 | 費用 | 寄稿者の負担 | 写真対応 | SPの手間 | 自動化拡張性 | 判定 |
|------|------|-------------|---------|-----------|-------------|------|
| **Google Docs テンプレ** | 無料 | 低（普段使いのツール） | **◎ 本文中に直接配置** | Phase2で激減 | 高（GAS連携可能） | **推奨** |
| Googleフォーム + GAS | 無料 | 最小（URL1つ） | △ 添付のみ（位置指定が難しい） | Phase2で激減 | 高 | 写真なし記事なら○ |
| WPフォームプラグイン | 年$49〜199 | 低 | △ | 低 | 中 | コスト見合わず |
| LINE/DM（現状） | 無料 | 最小 | ○ | 高（全て手動転記） | 低 | 今は困ってない |

### Googleフォームを見送った理由

- 写真を「本文のどこに入れるか」がフォームでは指定しにくい
- 寄稿者は文章+写真をセットで構成したい → Docsの方が自然
- フォームは定型データ収集（チーム紹介等）には向いているが、記事原稿には不向き

## 推奨フロー: Google Docs テンプレート方式

### Phase 1（今は急がない → 寄稿者が増えたら実施）

```
① 寄稿テンプレート（Google Docs）を用意
   → 構成: タイトル / カテゴリ / 本文 / 写真挿入済み / 掲載希望日

② 寄稿者がテンプレをコピーして記入
   → 写真は本文中の好きな場所に直接貼り付け

③ 完成したら共有フォルダに移動 or SPにリンク共有

④ GAS が共有フォルダを監視 → 新ファイル検知 → LINE通知

⑤ SPがWPに投稿（手動）
```

**メリット:**
- 寄稿者にとって最も自然なワークフロー（Docsで書くだけ）
- 写真の配置が自由 → 完成形に近い原稿が届く
- Googleアカウントがあれば誰でも使える
- 無料

### Phase 2（余裕ができたら・2〜4時間）— GAS連携で半自動化

```
共有フォルダの新規Docs
  │
  ▼ GAS
Docs の内容を取得（テキスト + 画像）
  │
  ▼ GAS
WordPress REST API で下書き自動作成
  ├── POST /wp-json/wp/v2/posts（下書き投稿）
  ├── POST /wp-json/wp/v2/media（画像アップロード）
  └── 認証: アプリケーションパスワード（WP 5.6+）
  │
  ▼
LINE Notify で通知「○○さんから寄稿が届きました。下書きに保存済みです」
  │
  ▼
SPはスマホで確認 → 公開ボタンを押すだけ
```

**技術ポイント:**
- `DocumentApp.getBody()` で Docs の本文をHTML的に取得
- `InlineImage` で本文中の画像を抽出 → WP REST API でアップロード
- 画像の挿入位置も本文の順序から推定可能

### Phase 3（寄稿者10名超になったら）

- スプレッドシートにステータス管理（受付済 / 確認中 / 公開済 / 差し戻し）
- GASで公開時にステータス自動更新
- 寄稿者ごとの掲載実績ダッシュボード

## 現状（LINE/DM方式）を当面続ける判断について

**今すぐ変える必要はない。** 理由:
- 寄稿者3名+1で回っている
- 大きな不便は感じていない
- フローを変えること自体が寄稿者への負担になりうる

**切り替えのトリガー:**
- 寄稿者が5名を超えたとき
- 管理が追いつかなくなったとき
- 新しい寄稿者に「ここから出してね」と案内する必要が出たとき
