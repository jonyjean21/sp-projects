## コンセプト

「もう1人のSP」= **SPの記憶・判断・行動を再現できるAIシステム**

現状のClaude Codeは「毎回初対面の優秀な部下」。
目指すのは「SPの分身として自律的に考え、動き、学ぶ存在」。

## 層1: 記憶層 — sp-brain/

**目的**: セッションを跨いでSPの知識・判断・経験を蓄積する

### ディレクトリ構造

```
sp-brain/
├── identity/
│   ├── core-values.md          # SPの価値観・信念・大切にしていること
│   ├── decision-patterns.md    # 判断パターン（条件→結論のマッピング）
│   ├── communication-style.md  # 口癖・文体・コミュニケーションの癖
│   └── ng-lines.md             # 絶対にやらないこと・NGライン
│
├── knowledge/
│   ├── molkky/                 # モルック関連の専門知識
│   │   ├── rules-and-strategy.md
│   │   ├── community-map.md    # 人間関係・コミュニティ構造
│   │   └── hub-editorial.md    # MOLKKY HUB編集方針
│   ├── tech/                   # 技術的な知識・好み
│   │   ├── stack-preferences.md
│   │   └── architecture-decisions.md
│   └── business/               # ビジネス関連の知識
│       ├── revenue-strategy.md
│       └── partner-relations.md
│
├── memory/
│   ├── decisions/              # 過去の重要な意思決定ログ
│   │   └── YYYY-MM-DD-{topic}.md
│   ├── learnings/              # 「これは覚えておけ」の蓄積
│   │   └── YYYY-MM-DD-{lesson}.md
│   └── people/                 # 人物メモ（関係者の情報）
│       ├── naka-san.md
│       ├── shisho.md
│       └── hub-writers.md
│
└── context/
    ├── current-priorities.md   # 今の最優先事項（動的に更新）
    ├── active-projects.md      # 進行中プロジェクト一覧
    └── weekly-state.md         # 今週の状態・気分・フォーカス
```

### 記憶の更新ルール

1. **自動蓄積**: セッション終了時、重要な判断や学びがあれば `memory/` に追記
2. **SP指示**: 「これ覚えておいて」→ 適切な場所に保存
3. **定期棚卸**: 月1回、`context/` を最新化
4. **CLAUDE.mdから参照**: `sp-brain/identity/` は CLAUDE.md の `思考DNA` セクションから参照

### 記憶の読み込み方式

```
セッション開始
  → session-start.sh が sp-brain/context/ を要約表示
  → 必要に応じて sp-brain/identity/ を参照
  → 深い判断が必要な時のみ sp-brain/knowledge/ を掘る
```

**ポイント**: 全部読むと遅い＆コスト高なので、階層的に必要な分だけ読む

## 層2: 判断層 — AI組織の進化

既存の `ai-org-structure.md` をベースに、記憶層と接続。

### 変更点

| 現状 | 進化後 |
|------|--------|
| AI-SPは CLAUDE.md のルールだけで判断 | sp-brain/ の記憶も参照して判断 |
| 部長スキルは独立して動く | 部長→AI-SP→記憶の参照チェーン |
| 判断ログは残らない | 重要判断は `memory/decisions/` に自動保存 |
| セッション毎にリセット | 前回の文脈を `context/` から復元 |

### 判断フローの変化

```
Before:
  SP指示 → Claude Code → 実行 → 結果

After:
  SP指示（or 外部トリガー）
    → AI-SP が sp-brain/identity/ で判断基準を確認
    → sp-brain/context/ で今の状況を把握
    → 部長AIに振り分け
    → 実行結果を sp-brain/memory/ に蓄積
    → 次回のセッションで「前回こうした」が分かる
```

## 層3: 接続層 — 外部世界とのインターフェース

### 3-A: LINE Bot（メインの対話チャネル）

SPが外出中でもLINEで「もう1人のSP」に指示・相談できる。

```
┌──────────┐    Webhook     ┌──────────────┐    Claude API    ┌──────────┐
│ LINE App │ ──────────────→│ Vercel API   │ ───────────────→ │ Claude   │
│ (SP)     │←──────────────│ /api/line-bot │←───────────────│ Sonnet   │
└──────────┘    Reply       └──────┬───────┘                 └──────────┘
                                   │
                                   ▼
                            ┌──────────────┐
                            │ Firebase     │
                            │ /line-logs   │
                            │ (会話履歴蓄積) │
                            └──────────────┘
```

**実装計画**:

| コンポーネント | 技術 | コスト |
|-------------|------|--------|
| LINE Messaging API | 無料プラン（月200通） | ¥0 |
| Webhook受信 | Vercel Serverless `/api/line-bot.js` | ¥0（既存プラン内） |
| AI処理 | Claude API (Sonnet) | ¥0（サブスク内 or 従量） |
| 会話ログ | Firebase `/line-logs` | ¥0（既存DB内） |
| **合計** | | **¥0** |

**LINE Botの機能**:

```
SP「今週のタスク教えて」
  → tasks/current.md を読んで要約返信

SP「記事のアイデア3つ出して」
  → sp-brain/knowledge/molkky/ を参照
  → コンテンツ部長スキルで企画
  → LINEに3案返信

SP「大会情報の処理状況は？」
  → Firebase /tournament-queue を確認
  → 処理済み/未処理の件数を返信

SP「これ覚えておいて：〇〇は△△が好き」
  → sp-brain/memory/ に保存
  → 「覚えました」と返信
```

## Obsidian / Antigravity の位置づけ

### Obsidian
- **結論**: sp-brain/ をMarkdownで管理する方が良い
- 理由: Claude Codeから直接読み書きできる。Obsidianは人間が読む時のビューアとして使うだけ
- やるなら: sp-brain/ をObsidian Vault として開けるようにしておく（追加作業ゼロ）

### Antigravity
- **結論**: Phase 3が安定してから検討
- 理由: GitHub Actions + Vercel Cron で無料でできる範囲が広い
- Antigravityが必要になるケース: Claude Code自体を24時間常駐させたい場合（Phase 4以降）

## FAQ

**Q: 記憶が増えすぎたらコンテキストウィンドウに入らないのでは？**
A: 階層的に読む設計にしている。全部読まない。`context/`（数KB）→ 必要なら `identity/`（数KB）→ 深い判断の時だけ `knowledge/`。

**Q: LINE Botの無料枠（月200通）で足りる？**
A: SPの使い方なら十分。超えたら月¥5,000のライトプランに上げるが、まず無料で始める。

**Q: sp-brain/ をGitに入れて大丈夫？**
A: publicリポジトリなので、個人情報・機密情報は入れない。人物メモ等は `.gitignore` に追加するか、Firebase に移す。

**Q: 既存のAI組織(ai-org-structure.md)との関係は？**
A: AI組織は「判断層」の設計。このドキュメントはそれに「記憶層」と「接続層」を追加した上位設計。
