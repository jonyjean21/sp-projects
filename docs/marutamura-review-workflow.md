# マルタ村アプリ コードレビュー・ワークフロー

## 概要

中さんがManusで作ったアプリを、SP（+Claude）がレビューし、
フィードバックを返して品質を上げていくサイクル。

## フロー

```
① 中さん: Manusでアプリ開発
② 中さん: GitHubにpush（marutamura org）
③ SP: Claude Codeでコードレビュー実施
④ SP: GitHub Issueにフィードバックを起票
⑤ 中さん: 修正してManusで再デプロイ
⑥ 繰り返し
```

## レビュー方法（Claude Codeでの実施手順）

### 前提

- marutamuraのリポジトリにアクセスできるセッションが必要
- または、リポジトリをローカルにcloneしてからレビュー

### レビュー実施コマンド例

```
「marutamura/oshi-katsuのコードレビューをして。
セキュリティ、パフォーマンス、コード品質の観点で。
見つけた問題はGitHub Issueに起票して。」
```

### 代替案: sp-projectsにレビュー結果を記録

GitHub Issueに直接起票できない場合は、
`docs/reviews/` ディレクトリにレビュー結果をmdで残し、
チャプチェ会で中さんに共有する。

## レビュー観点チェックリスト

### セキュリティ（最優先）
- [ ] 認証・認可の実装は適切か（JWT、SSO）
- [ ] SQLインジェクション対策（Drizzle ORM使用なら基本OK）
- [ ] XSS対策（ユーザー入力のサニタイズ）
- [ ] CSRF対策
- [ ] 環境変数・シークレットがハードコードされていないか
- [ ] APIエンドポイントの権限チェック（admin only等）

### パフォーマンス
- [ ] 不要なre-renderがないか
- [ ] N+1クエリがないか
- [ ] 画像最適化
- [ ] バンドルサイズ

### コード品質
- [ ] TypeScript型安全性（any乱用がないか）
- [ ] エラーハンドリング
- [ ] 重複コード
- [ ] テストカバレッジ

### UX
- [ ] モバイル対応（レスポンシブ）
- [ ] ローディング状態の表示
- [ ] エラー時のユーザーフィードバック

## レビュー優先順位

| 1 | oshi-katsu | 認証あり（JWT+SSO）、招待制 → セキュリティ重要 |
