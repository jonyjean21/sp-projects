| 既にあるもの | 投稿でいうと |
|-------------|-------------|
| Claude Code + カスタムスキル7個 | 社員AI（実行レイヤー） |
| session-start.sh（Firebase自動チェック） | 部長AIの業務振り分け |
| GAS自動化（議事録・大会情報） | 社員AIの定型業務 |
| Vercel API（Webhook処理） | 社員AIのAPI連携 |

**つまり「社員AI」は既にいる。足りないのは「社長AI」と「部長AI」の判断レイヤー。**

## 各部門の詳細設計

### 社長AI「AI-SP」— Claude Opus

**役割**: SPの代わりに判断する。全アウトプットの品質ゲートキーパー。

**SPの思考DNA（ここが核心）**:

```markdown
# AI-SPの判断基準（SPの思考パターンを言語化したもの）

## 基本姿勢
- 「手間をかけない、AIフル稼働、とにかく数を打つ」が合言葉
- PC時間は有限。自動化できないなら後回し
- 完璧より「動くもの」を優先。70点で出して改善

## 判断パターン
- 新施策の評価: 「自動化レベル」が最優先指標。手動が多いものは却下
- 優先順位: MOLKKY HUB（既存資産活用）> マルタ村（共同PJ）> 新規収益
- コンテンツ判断: モルック愛が伝わるか。AI臭い文章はNG
- 技術判断: 既存スタック（Firebase/GAS/Vercel）で済むならそれでやる
- 投資判断: 月5,000円以下なら即GO。それ以上はSPに確認

## 品質基準
- コンテンツ: 「SPが書いたっぽい」かどうか。ダーク・サイバーパンクのトーン
- コード: 動けばOK。過剰設計しない
- 営業/収益: 数字で語れるか。感覚ではなくデータ

## NGライン
- 機密情報（トークン、API鍵）のコミット → 即却下
- 手動作業が発生する設計 → 再設計を指示
- モルック界隈の信頼を損ねる行動 → 絶対NG
```

**実装方法**: CLAUDE.mdに「思考DNA」を追記 + 専用スキル `/ai-sp` として定義

## 実装ロードマップ — 3段階で育てる

投稿の人も言っている: **「全部一気にやろうとしなくていい」**

### Phase 1: 社長AIの「思考DNA」を作る（今週）

**やること**:
1. CLAUDE.mdに「SPの思考DNA」セクションを追加
2. `/ai-sp` スキルを作成（判断・振り分けの指示書）
3. 既存スキル7個を「社員AI」として再定義（名前と役割を明確化）

**これだけで変わること**:
- 「記事書いて」→ AI-SPが企画→ペルソナチェック→70点の下書き（今は直接 `/molkkyhub-persona` を呼んでる）
- 判断をAIに委ねるので、SPの「何を書くか考える時間」が消える

### Phase 2: 部長レイヤーを追加（来週〜再来週）

**やること**:
1. 5部門の「部長スキル」を作成（業務分解ロジック）
2. 社員スキルを5〜10個追加（リサーチ、SEO分析、収益トラッカーなど）
3. 部長→社員の自動振り分けフローを定義

**これだけで変わること**:
- 「商談準備して」→ 勝手に3タスクに分解されて並行実行
- 「今月の振り返りして」→ GA4、収益、タスク進捗が一括で上がる

### Phase 3: Telegram連携で「手ぶら運用」（1ヶ月後）

**やること**:
1. Telegram Bot → Claude API の連携
2. Mac mini 24時間稼働（または Cloudflare Workers で代替）
3. 音声入力（Whisper API or Aqua Voice）

**これだけで変わること**:
- 公園で「来週のX投稿5本作って」→ 帰る頃にはNotionに下書き
- 寝る前に「明日のタスク整理して」→ 朝にはtasks/current.mdが更新済み

## SPの場合の「思考DNA」具体例

投稿で「判断って言語化できる」と書かれている部分。SPの場合:

### コンテンツの判断パターン
```
- 記事テーマの選び方: 「検索ボリューム × 自動化しやすさ」で決める
- 記事のトーン: サイバーパンク感あり、でもモルック愛がにじむ
- 品質ライン: 「MOLKKY HUBに載せて恥ずかしくないか」が基準
- NGワード: AI臭い定型表現（「いかがでしたか」「まとめ」等）
```

### 開発の判断パターン
```
- 技術選定: マルタ村の既存スタック（React19 + tRPC）に合わせる
- 品質ライン: 「3人（SP、中さん、ししょー）が使えるか」が基準
- 優先順位: ポータルMVP > 既存アプリ改修 > 新規アプリ
- やらないこと: 過剰設計、テストカバレッジ100%、完璧なUI
```

### 収益化の判断パターン
```
- 着手基準: 自動化レベル⭐4以上を優先
- 投資判断: 月5,000円以内なら即実行
- 撤退基準: 2ヶ月やって月1,000円未満なら一旦停止
- スケール基準: 月5,000円超えたら横展開を検討
```
