# Claude Codeに"引き継ぎ"させたら、寝てる間も仕事が進んでた話

ターミナルを閉じると、AIは全部忘れる。

Claude Codeを使い始めて最初にぶつかった壁がこれだった。1時間かけて仕様を説明して、コードを書かせて、デバッグして、いい感じに動き始めたところでPCがスリープ。次にターミナルを開いたら「何のプロジェクトですか？」と聞かれる。

毎回最初から説明するのか？ それじゃ人間が作業した方が速い。

この問題を解決するために作ったのが **SESSION-HANDOFF.md** という仕組みだ。

---

## 概要：AIに「引き継ぎ書」を書かせる

SESSION-HANDOFF.mdは、要するに**AIが次の自分（次のセッション）に向けて書く引き継ぎ書**だ。

人間の仕事で言えば、夜勤と日勤の引き継ぎに近い。「ここまでやった」「次はこれをやる」「注意点はこれ」を1枚のドキュメントに残す。これをAIにやらせる。

仕組みはシンプルだ。

1. 作業セッションの最後に「SESSION-HANDOFF.mdを更新して」と言う
2. Claude Codeが今日やったこと・次にやること・注意点をMarkdownに書き出す
3. 次のセッション開始時に「SESSION-HANDOFF.mdを読んで」と言う
4. Claude Codeが即座に文脈を復元して作業を再開する

これだけで、ターミナルを閉じてもAIが文脈を失わなくなった。

実際にこの仕組みを導入してから、セッション開始時の「状況説明タイム」がゼロになった。以前は毎回10〜15分かけてプロジェクトの状況を説明していたのが、今は一言で済む。

## 何が変わったか

導入前と導入後で、こう変わった。

**導入前：**
- セッション開始に15分のコンテキスト説明が必要
- 前回やったことを自分が覚えていないと詰む
- 複数プロジェクトを並行すると混乱する
- PCスリープ = 作業リセット

**導入後：**
- セッション開始は「引き継ぎ読んで」の一言で完了
- 前回の作業内容はAIが記録しているから自分が覚える必要なし
- 10個以上のプロジェクトを並行しても混乱しない
- 寝ている間にcronで自動実行 → 朝起きたら結果が出ている

特に大きかったのは「寝ている間にcronで自動実行」の部分だ。SESSION-HANDOFF.mdに「次にやること」が書いてあるから、cronジョブに「SESSION-HANDOFF.mdを読んで次のタスクを実行しろ」と設定すれば、寝ている間もAIが仕事を進めてくれる。

## ファイル構成：SESSION-HANDOFF.mdの置き場所

SESSION-HANDOFF.mdはプロジェクトのルートに置く。理由は、Claude Codeがセッション開始時に最初にアクセスできる場所にする必要があるからだ。

```
my-project/
├── CLAUDE.md              ← プロジェクト全体のルール
├── SESSION-HANDOFF.md     ← セッション間の引き継ぎ（これ）
├── .claude/
│   └── memory/
│       └── MEMORY.md      ← 永続的な記憶（プロジェクト知識）
├── src/                   ← ソースコード
└── docs/                  ← ドキュメント
```

重要なのは **CLAUDE.md** と **MEMORY.md** との使い分けだ。

| ファイル | 役割 | 更新頻度 | 内容 |
|---------|------|---------|------|
| CLAUDE.md | プロジェクトのルール・方針 | 月1回程度 | コーディング規約、運用ルール |
| MEMORY.md | 永続的な知識・状態 | 週1回程度 | アーキテクチャ、技術スタック、進行中タスク |
| SESSION-HANDOFF.md | セッション間の引き継ぎ | 毎セッション | 今日やったこと、次やること、注意点 |

この3層構造が肝だ。CLAUDE.mdは「憲法」、MEMORY.mdは「長期記憶」、SESSION-HANDOFF.mdは「短期記憶」と考えるとわかりやすい。

## テンプレートの各セクション解説

### 「ミッション」セクション

ここにはプロジェクトの最上位目標を1行で書く。Claude Codeは文脈が長くなると目的を見失うことがある。冒頭にミッションを明記しておくと、判断に迷ったときの指針になる。

俺の場合は「3月中に¥100,000を稼ぐ」と書いている。タスクの優先順位を判断するとき、AIがここを参照して「収益に直結する作業を優先する」という判断を自律的にやるようになった。

### 「完了済み」セクション

前回のセッションでやったことをテーブル形式で書く。ファイルパスを必ず添える。AIは「何をやったか」より「どこにあるか」を知りたい。ファイルパスがあれば中身を読んで文脈を復元できる。

### 「次にやること」セクション

これが最も重要なセクション。ここのクオリティで次のセッションの生産性が決まる。

ポイントは3つ。

1. **優先順位を明記する** — 番号付きで上から順に
2. **完了条件を書く** — 「何ができたらOKか」がないと、AIが延々と作業を続ける
3. **見込み時間を書く** — AIの時間見積もりは甘いが、書くこと自体がスコープの制御になる

### 「注意事項・地雷」セクション

これがないとAIは同じミスを繰り返す。前回踏んだバグ、環境固有の制約、やってはいけない操作をここに書く。

例えば俺の環境では「SiteGuard WAFが有効だからPUT/DELETEは使うな、POSTだけ使え」という注意がある。これを書かなかった初期は、毎回403エラーで10分無駄にしていた。

### 「SPがやること」セクション

AIにはできない作業を明示する。ブラウザでの手動操作、支払い、外部サービスの認証など。これを書いておくと、AIが「この作業はSPに任せる」と判断して、自分の担当範囲に集中する。

## 応用テクニック

### 応用1: 複数プロジェクトの並行管理

俺は今、10個以上のプロジェクトを同時に動かしている。各プロジェクトにSESSION-HANDOFF.mdを置いて、メインのMEMORY.mdに全プロジェクトの状態サマリーを書いている。

```
MEMORY.md（メインの永続記憶）
├── プロジェクトA: SESSION-HANDOFF.md → 進行中、次はXXをやる
├── プロジェクトB: SESSION-HANDOFF.md → 待ち状態、SP確認後に再開
└── プロジェクトC: SESSION-HANDOFF.md → 完了、リリース済み
```

セッション開始時に「MEMORY.mdを読んで、今日はどのプロジェクトを優先すべきか提案して」と聞くと、AIが全プロジェクトの状態を把握した上で「今日はプロジェクトAを優先すべき。理由は〜」と提案してくれる。

### 応用2: チーム開発での引き継ぎ

個人で使っている仕組みだが、チーム開発にも応用できる。

SESSION-HANDOFF.mdをGitにコミットして共有すれば、チームメンバー全員がプロジェクトの最新状態を把握できる。特にClaude Codeを複数人で使うチームの場合、「前の人がどこまでやったか」がファイルを見るだけで分かる。

具体的には、ブランチごとにSESSION-HANDOFF.mdを更新するルールを作る。PRをマージするとき、SESSION-HANDOFF.mdの「完了済み」セクションも更新する。これでmainブランチのSESSION-HANDOFF.mdが常にプロジェクトの最新状態を反映する。

### 応用3: cronとの組み合わせで夜間自動実行

SESSION-HANDOFF.mdの「次にやること」セクションをcronジョブから読み取らせる。

```bash
# crontab に設定
0 3 * * * cd /path/to/project && claude "SESSION-HANDOFF.mdを読んで、次のタスクの1番を実行して。完了したらSESSION-HANDOFF.mdを更新して。"
```

午前3時にAIが起動して、タスク1を実行して、SESSION-HANDOFF.mdを更新して終了する。朝起きたら結果が出ている。

ただし自動実行には注意が必要だ。AIが判断を間違えたとき、自動で修正するのは危険だ。最初は `--dry-run` オプション付きで動かして、結果をログに残すだけにする。信頼できるタスクだけを自動実行に回す。

### 応用4: 日次レポートの自動生成

SESSION-HANDOFF.mdの更新履歴をgit logから集計すれば、日次・週次レポートが自動生成できる。

「今週のSESSION-HANDOFF.mdの変更履歴を見て、今週やったことのサマリーを作って」

これだけで週報が完成する。上司やクライアントへの報告書にそのまま使える形式で出力させることもできる。
